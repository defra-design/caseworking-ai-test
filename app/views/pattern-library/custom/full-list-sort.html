Full-list column sorting

Sortable table columns that re-order the whole list across every page, not just the rows currently visible. A server-side variant of the MOJ sortable table, used on the caselist where the results are paginated.

The MOJ sortable table sorts in the browser, so on a paginated list it only re-orders the 20 rows on the current page — the "oldest" case could be on page 7 and never move. This pattern instead makes each column heading a link; the route sorts the entire filtered set and re-paginates, so page 1 always shows the true top of the chosen order.

How it works

  1. Each sortable heading is a link to the same tab with ?sort=<column>&dir=<asc|desc>. Clicking the already-active column toggles the direction.
  2. The route reads sort / dir, sorts the full filtered set (dates by real date value, everything else case-insensitive text), then paginates the sorted result.
  3. The active column carries aria-sort="ascending|descending"; the rest are aria-sort="none". The list loads pre-sorted (the caselist defaults to Submitted, oldest first — "by age").
  4. Pagination links carry the current sort / dir so paging keeps the order; choosing a new column resets to page 1.

Example

The Submitted column is the active sort (ascending); the others are inactive links.

<!-- Route: sort the WHOLE filtered set, then paginate. -->
const sorted = sortRows(filtered, sort, dir)   // dates by date value, else text
const view   = paginate(sorted, page)          // page links keep &sort=&dir=

When to use

  • Any list that is paginated server-side and that users sort — the caselist is the main case.
  • When the correct top-of-list item could be on any page, so client-side (visible-page) sorting would mislead.

When not to use

  • When every row the user will sort is already on one page — the in-browser MOJ sortable table is simpler and needs no reload.
  • For re-ordering that should not change the URL or trigger a page load.

Customisation in this prototype

  • Headings render from the sortHeader macro in app/views/Grasslands/includes/_caselist-macros.html.
  • The sort + paginate logic (grassSortRows, grassSortState, grassBuildView) lives in routes.js; dates sort on a parsed date value so "by age" is true chronological order.
  • No new CSS — links reuse govuk-link; aria-sort on the <th> drives the GDS/MOJ sort indicator.

Where does this occur

  • Grasslands caselist — the My cases, team context and Completed tabs (each defaults to Submitted, oldest first).
