Team caselist Experimental

A data-driven extension of the Caselist for teams: a caseworker sees their own work, can switch the view to any team or to everyone, and reads completed cases in a separate tab. Built so the whole list is driven by a data file and the team split is held in one place.

It adds three things to the basic caselist:

  • a context selector above the tabs that switches the view between each team and "All cases";
  • a My cases / context / Completed cases tab structure, where the middle tab follows the selected context and completed work is split out; and
  • data-file backing — cases and teams live in data files, so the routes filter and paginate and the templates just render.

When to use this

  • A caseworker queue that is organised into teams and you need to move between "my work", "my team's work" and "the whole queue".
  • Lists large enough to need pagination and a clear separation of in-progress vs completed cases.
  • When the case data should be generated/maintained in one place rather than hand-written into the page.

When not to use this

  • A single, short, unsegmented list — use the simpler Caselist pattern.
  • There are no teams and no completed/active distinction to make.

Context selector

A govuk-select ("Switch team") listing each team plus an All cases option, with a Switch button, sitting above the tabs. Choosing a context and pressing Switch reloads the page; the choice is held in the session so it carries across tab clicks. The selected context drives both the middle (context) tab and the Completed tab.

Tabs

  • My cases — the current user's own cases, independent of the selected context.
  • Context tab — the selected team's cases, or all cases when the context is "All cases". Its label is the current context (for example "Team A" or "All cases").
  • Completed cases — cases in a final outcome (Agreement accepted, Rejected or Withdrawn), filtered by the same context.

Tabs are sibling pages with tab-style navigation (each non-active tab is a link), following the MOJ-style multi-page tab approach already used elsewhere in this prototype.

Active vs completed

Completed cases (Agreement accepted / Rejected / Withdrawn — see status tags) appear only on the Completed tab. My cases and the context tab show active cases only, so a caseworker's working lists are not cluttered by finished work.

Filters, pagination and multi-assign

  • Search + filters in a details ("Find an application"), modelled on the calculations filter / MOJ filter.
  • Pagination using the MOJ pagination component on the context and Completed tabs.
  • Multi-assign — a Select checkbox per row plus a "Reassign selected cases" button, as in the FG-casework caselist.
  • A sortable table with the columns Select · ID · Business · SBI · Submitted · Status · Assignee.

How it is built

The pattern is route-driven and data-file backed:

  • data/<area>-teams.js — the teams and their caseworkers (single source of the split).
  • data/<area>-cases.js — the case records (one per row), each tagged with a team and an assignee.
  • Routes filter (by tab + context) and paginate, then pass a ready-made view to thin templates; the selected context is held in the session.

The repeatable build is captured in the data-driven-caselist skill; the Grasslands caselist is the reference implementation.

Accessibility

  • The context selector is a labelled govuk-select with an explicit Switch button — no on-change auto-submit, so keyboard and screen-reader users stay in control.
  • Status is conveyed by the tag's text, not colour alone.
  • Pagination exposes the result count ("Showing X to Y of Z") and marks the current page; the table is a real sortable table.
  • Row select checkboxes have visually-hidden labels naming the case.

Where does this occur

  • Grasslands — My cases
  • Grasslands — context tab (Team / All cases)
  • Grasslands — Completed cases
