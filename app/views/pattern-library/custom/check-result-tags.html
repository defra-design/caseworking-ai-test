Check-result tags

The tags shown in the summary row of every rule check on the calculations pages, so a caseworker can read each outcome without opening the check.

Three tags cover the outcomes these checks produce:

  • Passed / Failed — for checks with a regulatory pass-or-fail outcome (Available area check, Moorland threshold check). A green tag with a tick for a pass, a soft-red tag with a cross for a fail.
  • Checked — for checks that confirm something has been looked at rather than passing or failing (the SSSI / HEFER checks once they're done). Same green tick as a pass, with the label "Checked".
  • Action — for the same SSSI / HEFER checks while they still need follow-up work: a pale-blue pill with a leading "!". These checks have no pass/fail outcome, so they flag a task rather than a result.

Pass

Green govuk-tag--green with an inline .tick-icon svg. The svg is included inline so the icon scales and inherits colour cleanly with the tag.

Example
Passed 
<strong class="govuk-tag govuk-tag--green">
  <svg class="tick-icon" width="20" height="20" viewBox="0 0 24 24"
       fill="green" xmlns="http://www.w3.org/2000/svg">
    <path d="M9 16.2L4.8 12L3.4 13.4L9 19L21 7L19.6 5.6L9 16.2Z"/>
  </svg>
  Passed
</strong>

Fail

Custom govuk-tag--red-new — a softer red background with dark red text, plus a cross icon rendered via a ::before pseudo-element pointing at /public/images/icon-tag-remove-cross.svg. No inline svg is required.

This is the same soft-red palette as the "Not accepted" case status tag (govuk-tag--red-status) — the two share their colour values via the tokens in _tags.scss. The difference is the cross icon: the Fail check-result tag carries it, the case-status tag does not. So they read as one colour with two roles — Pass/Fail for individual checks, status words ("Not accepted") for the whole case — but they are kept as separate classes precisely because of that icon. Use govuk-tag--red-new for a failed check; use govuk-tag--red-status for case status.

Example
Failed 
/* application.scss */
.govuk-tag--red-new {
  color: rgb(42.4, 10.6, 5.6);
  background-color: rgb(244.25, 204.5, 198.25);
}
.govuk-tag--red-new:before {
  background-image: url(/public/images/icon-tag-remove-cross.svg);
  content: "";
  display: inline-block;
  height: 10px;
  width: 10px;
  margin-left: 5px;
  margin-right: 10px;
  vertical-align: middle;
}

<!-- Usage -->
<strong class="govuk-tag govuk-tag--red-new">Failed</strong>
Migration note. govuk-tag--red-new and govuk-tag--red-status resolve to the same colour tokens, and _tags.scss already marks --red-new as a deprecated alias. They can't simply be merged, though: only --red-new carries the cross-icon ::before, and the case-status tag must not show a cross. A real consolidation would move the cross onto a check-specific class (not the colour) first. Until then, keep using --red-new for failed checks — every live calculations screen already does.

Checked

Same green-tick visual as Pass, with the label swapped to Checked. Used on the SSSI / HEFER checks (e.g. Is the site of special scientific interest?) once the caseworker has dealt with them — these checks confirm something has been looked at rather than passing or failing.

Example
Checked 
<strong class="govuk-tag govuk-tag--green">
  <svg class="tick-icon" width="20" height="20" viewBox="0 0 24 24"
       fill="green" xmlns="http://www.w3.org/2000/svg">
    <path d="M9 16.2L4.8 12L3.4 13.4L9 19L21 7L19.6 5.6L9 16.2Z"/>
  </svg>
  Checked
</strong>

Action

Custom govuk-tag--action — a pale-blue pill with a leading bold "!" (added via a ::before). Used on the same SSSI / HEFER checks while they still need follow-up work (request a consent notice, raise a HEFER). These checks have no pass/fail outcome, so the tag flags a task to do rather than a result. The "!" is the only non-text affordance — the word "Action" still carries the meaning.

Example
Action 
/* application.scss — the "!" is drawn by the ::before */
.govuk-tag--action {
  color: #0f385c;
  background-color: #d2e2f1;
  display: inline-flex;
  align-items: center;
}
.govuk-tag--action::before {
  content: "!";
  margin-right: 8px;
}

<!-- Usage -->
<strong class="govuk-tag govuk-tag--action">Action</strong>

All four side by side

Outcome Tag When used
Pass Passed Regulatory pass/fail checks — Available area, Moorland
Fail Failed Same checks, when the threshold is not met
Checked Checked SSSI / HEFER checks once dealt with — no pass/fail outcome
Action Action Same SSSI / HEFER checks while follow-up work is outstanding

Layout inside the check summary

When used inside a <summary> row, the application-level rule on .govuk-details__summary-text floats the tag to the right edge of the row with a 10px gap, so it aligns regardless of the check name's length — see the Details pattern.

When to use

  • To show the outcome of a single, atomic rule check at the end of its summary row.
  • To mark a check that confirms something has been looked at (Checked) or still needs follow-up (Action) rather than passing or failing.

When not to use

  • For overall case status (use the colour tags documented on the Status tags page).
  • For task-list status indicators (use the standard Task list tags).
  • To convey severity beyond these states — Passed/Failed/Checked/Action cover a pass, a fail, a completed informational check and an outstanding one; if a check needs a different state (for example "in progress"), pick a different tag rather than overloading these.

Where does this occur

  • Real-case calculations (Golden Grange) — Passed / Failed on the Available area and Moorland checks; Checked or Action on each SSSI / HEFER check depending on whether it's been dealt with.
  • Large calculations (101 parcels)
  • Medium calculations
  • Small calculations
