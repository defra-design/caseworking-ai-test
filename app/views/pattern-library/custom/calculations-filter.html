Calculations filter

A custom variant of the MOJ filter component used on every calculations page in this prototype to swap between rule-check views and switch between rules-check runs.

It departs from the canonical MOJ filter in two ways: there is no Selected filters tag list (the user can only have one filter active at a time), and the criteria are radio buttons rather than checkboxes. Selecting a radio and clicking Apply filters swaps the rendered list of checks in place; switching the Run date dropdown reloads the page at the corresponding historical run with the selected filter carried through.

Differences from the standard MOJ filter

  • Radios instead of checkboxes. The Listing section is mutually exclusive — a check is either visible to the caseworker or not — so radios match the semantics better than checkboxes and remove the "no selection" ambiguity.
  • No selected-filters tag list. With only one active filter at a time and three obvious options, the tag list adds noise without helping. The currently-selected radio is the indicator of state.
  • No "Clear filters" link. Clearing is equivalent to selecting All checks, which is one click away.
  • Run-date dropdown doubles as a version switcher. Each option's value is a page-key (e.g. case2-new2). Submitting the form with a different run selected redirects to that historical run, with the chosen Listing radio applied to the destination.

When to use

  • A long, scannable list of items where the caseworker needs to focus on a subset (here: fails or just changes).
  • The filtering is mutually exclusive and the subset is small (two or three options).
  • The page also represents one of several versions or runs of the same data set, and the user should be able to swap version while staying inside their current filter view.

When not to use

  • The user genuinely needs to combine filters (multiple statuses, multiple assignees, …) — use the standard MOJ filter with checkboxes and the selected-filters tag list.
  • The list of options is short enough to render without filtering — no filter panel is needed at all.

Per-page filter state

Filter state is scoped to each calc page independently (kept in req.session.data.calcFilters[<page-key>] by the /largeCalcFilter route handler). Visiting a sibling calc page renders that page's own filter state — there is no leakage between pages. The handler writes through to both req.session.data and res.locals.data so the upcoming Nunjucks render sees the swapped state.

Where does this occur

Every large or versioned calculations page in this prototype:

  • Large calculations (101 parcels)
  • Large calculations — middle run
  • Large calculations — baseline
  • Medium calculations
  • Medium calculations — previous run
  • Real-case calculations (Golden Grange)
  • Real-case calculations — middle run
  • Real-case calculations — baseline
