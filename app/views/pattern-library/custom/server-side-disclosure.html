Server-side disclosure

A "Show / Hide <thing>" link that reveals a secondary block of content by reloading the page with a query parameter, rather than by toggling it in the browser.

Used where a caselist carries useful-but-secondary context — budgeting figures, scheme metadata — that would otherwise push the cases themselves below the fold. The content is hidden by default and the caseworker opts in. A short-lived query parameter flips the state; the server stores it in the session, so it persists on its own across reloads, sorts, filters, pages and related tabs (see Keeping state across tabs) until the caseworker hides it again.

This is deliberately not the GDS details component. Details is a client-side disclosure for content already present in the page; this pattern is for content the server decides whether to render at all, and whose open/closed state needs to persist across navigation.

Hold the state in the session, not in the query string, whenever the disclosure sits above other session-backed controls such as filters. An earlier version rebuilt the toggle link by re-serialising the whole query string (?budget=yes); above a filter that coupled the two — applying a filter dropped the parameter and closed the panel, and re-serialising the multi-value filter params through URLSearchParams comma-joined them into one corrupt token that reset the list. A session flag keeps the disclosure and the filters completely independent.

The toggle link

A plain govuk-link whose text and target both flip with the state. The route computes the opposite-state href; the template only picks the verb. Keep the noun identical in both states ("Show budgeting information" / "Hide budgeting information") so the link doesn't appear to change meaning.

The route helper

The whole pattern is one small helper. It reads ?budget=show / =hide, stores the state on the session, and builds the href for the opposite state. Only the query-based caselist state (sort, dir, page) is carried on the link; the session-backed search and filters persist on their own and are deliberately not re-serialised, so toggling can never corrupt them.

routes.js

Returns showBudget and budgetToggleHref, merged into each caselist route's render context.

function grassBudgetToggle (req) {
  // Persist the open/closed state on the session, independent of the filters.
  const d = req.session.data || (req.session.data = {})
  if (req.query.budget === 'show') d.showBudgetGrass = true
  else if (req.query.budget === 'hide') d.showBudgetGrass = false
  const showBudget = !!d.showBudgetGrass
  // Carry only the query-based caselist state across the toggle. Filters/search
  // live in the session, so they must NOT be re-serialised here (that comma-joins
  // multi-value params and corrupts them).
  const keep = new URLSearchParams()
  ;['sort', 'dir', 'page'].forEach(function (k) {
    const v = req.query[k]
    if (v != null && v !== '') keep.set(k, v)
  })
  keep.set('budget', showBudget ? 'hide' : 'show')
  return {
    showBudget: showBudget,
    budgetToggleHref: req.path + '?' + keep.toString()
  }
}

// merged into the render context of every tab in the set
res.render('Grasslands/caselist', Object.assign({
  ctx: ctx, view: grassBuildView(rows, req)
}, grassBudgetToggle(req)))

Keeping state across tabs

Where the disclosure sits above a tab set, the open state must survive switching tab — otherwise opening the panel and moving tab silently closes it again, which reads as a bug. Because the state lives in the session, this comes for free: the tab links stay plain, and every tab reads the same flag.

Keeping the tabs in sync

When a disclosure sits above a tab set, every tab page must render it identically — a toggle that appears on one tab but not another is worse than no toggle at all. Put the back link, heading, toggle and disclosed block in a single shared include and pull it into each tab template, rather than copying the markup three times. In the Grasslands caselist this is Grasslands/includes/_caselist-header.html.

When to use

  • Secondary context on a working screen — summary figures, scheme metadata — that most users don't need on every visit but some need often.
  • When the open/closed state should survive a reload, a sort, a filter, or a move between related tabs.
  • When the block is expensive or noisy enough that you'd rather the server didn't render it unless asked.

When not to use

  • For content already in the page that just needs showing and hiding — use the GDS details component, which needs no round trip.
  • For anything the user must read to complete the task. A disclosure hidden by default will be missed; if it's essential, show it.
  • For many independent toggles on one page — each adds a query parameter and a full reload. Past two or three, reconsider the page's structure.
  • Inside a form with unsaved input — the reload discards it unless the form state is threaded through the query too.

Where does this occur

  • Grasslands caselist — My cases, Team cases and All cases — "Show / Hide budgeting information", revealing the budget allocation bar and the scheme metadata summary.
