Notes Ready

A single, running record of the notes a caseworker captures as they work a case. It builds up as decisions are made, is organised by stage, and each entry is reachable from the case history timeline so anyone can jump straight to the note behind an event.

When to use this

  • Anywhere a caseworker records the reasoning behind a decision or action that others may need to read later.
  • When notes accumulate across a multi-stage process (review → agreement → ongoing checks) and need to stay in one place.

When not to use this

  • For a one-off free-text field that is captured and shown on a single screen — a plain textarea is enough.
  • For a chronological record of events rather than the notes themselves — use the case history timeline.

How it works

  • It builds up. Each note row is only shown once the decision or action it relates to has been answered, so the page grows as the case progresses. Until then the stage shows an empty state ("There are no notes to display yet").
  • It is organised by stage. Notes are grouped under a heading for each stage of the case — Review, Agreement, 5 month check — each with its own table. New stages add new sections; they do not interleave.
  • It is linked from the timeline. Each timeline event that produced a note links to the relevant stage section via an anchor (for example notes#review), so users land in the right place rather than the top of the page.

Anatomy of a stage

  1. Stage heading — an <h2> naming the stage (and the anchor target for timeline links).
  2. Notes table — a sortable table with four columns:
    • Date — when the note was added (default sort, newest first).
    • Progress — the decision or action that prompted the note (e.g. "Review decision: Accepted").
    • Note — the free-text note itself.
    • Added by — the caseworker.
  3. Empty state — shown in place of the table until the stage has its first note.

Linking from the timeline

Every timeline event that produced a note links back to its stage section in the notes page, using a stage anchor. This keeps the two patterns in step: the timeline says what happened and when; the note says why.

<a class="govuk-link" href="notes#review">View note</a>

Accessibility

  • Each stage has a real heading, so the page can be navigated by heading and the anchors are meaningful landing points.
  • Sortable headers are buttons with aria-sort — keyboard operable and announced. See sortable table.
  • The empty state is real text in the DOM, so screen-reader users are told a stage has no notes yet rather than meeting an empty table.

Where does this occur

  • Notes — the running notes record, reached from the case tabs and from timeline events.
