Task Ready

A single unit of work a caseworker completes within a stage. Every task shares a formalised structure so the experience is consistent across the whole case: a title, a rich-text summary, a configured set of outcomes, and a compulsory note. Choosing an outcome updates the task's status in the task list and writes a note.

Anatomy

  1. Title — an <h1> naming the task (for example "Customer details review").
  2. Summary — a rich-text area that tells the caseworker what to check and do. It accepts a fixed subset of GDS text styles and components (see below).
  3. Outcomes — a set of radio options, set by configuration and identical for every task in the stage.
  4. Compulsory note — selecting an outcome reveals an "Explain this outcome" field that must be completed before the task can be confirmed.
  5. Confirm — records the outcome, which changes the task's status in the task list and creates the note.

What the summary can contain

The summary is rich text, but deliberately not the full GDS toolkit — it is for reading, not data capture or layout. Authors may use:

  • Headings — govuk-heading-m and govuk-heading-s
  • Body text — govuk-body and lead govuk-body-l, with bold for emphasis
  • Bulleted lists — govuk-list govuk-list--bullet
  • Numbered lists — govuk-list govuk-list--number (for ordered steps)
  • Links — govuk-link
  • Inset text — for a related note or constraint
  • Details — to tuck away supporting guidance

It must not contain form inputs, buttons, tables, tabs, accordions or images — anything interactive or structural. Those belong to the outcome controls or to a dedicated screen, not the summary. Warning text is also out of scope here; the compulsory-note requirement already enforces the important action.

Outcomes are set by stage

The outcome options are configuration, not per-task content. Every task in a stage offers the same outcomes, so a caseworker always recognises the choices. Different stages define different sets, for example:

  • Review stage: Accept · Request information from customer · Pause for internal investigation · Cannot accept
  • 5 month check stage: No action needed · Request information · Escalate · Cannot accept

What happens on confirm

  • The task status changes in the task list for the stage (for example from "To do" to "Completed", or to a state that reflects the chosen outcome such as "Information requested").
  • A note is created from the compulsory explanation. It appears in the task's own Outcome history and in the case Notes record, under the relevant stage, and is reachable from the timeline.

Accessibility

  • The outcomes are radios with conditional reveal — see the progressive-reveal guidance. The revealed note is associated with its outcome via data-aria-controls.
  • The note is mandatory: validate it on submit and, on error, show an error summary linked to the field.
  • Keep the summary as real headings, lists and links so it can be navigated and read by assistive technology.
