Caselist Ready

A page-level pattern for showing a caseworker the cases they need to act on, and letting them find a specific case. It combines a filter, a sortable table, status tags and pagination.

For a team-based queue with a My / team / All context switch and a separate Completed tab, see the Team caselist variant.

When to use this

  • The landing page for a caseworker — "my cases", "my team", "all cases", "new".
  • Any list of records the user scans, sorts and filters to pick one to work on.

When not to use this

  • For a single record's details — use the case's task list (the tasklist-stage screen).
  • For a short, fixed list that never needs filtering or sorting — a plain table is enough.

Anatomy

  1. Caselist tabs — switch between My cases / My team / All / New (GOV.UK secondary navigation or links).
  2. Filter — the MOJ filter, with selected filters shown as removable tags.
  3. Results count — "Showing 1 to 20 of 134 cases".
  4. Sortable table — the MOJ sortable table with a status tag column.
  5. Pagination — the MOJ pagination.

Content rules

  • The reference number is the link to the case — keep it the first column.
  • One status per row, using the documented status colours. Don't invent new colours per caselist.
  • Dates use the GOV.UK format (2 May 2026), and the column sorts by true date, not text.
  • Default sort is most-recently-received first.

Accessibility

  • Sortable headers are buttons with aria-sort — keyboard operable and announced. See sortable table.
  • The results count is in the DOM so screen readers can read how many results a filter returned.
  • Status is conveyed by the tag's text, not colour alone.
  • Pagination links have descriptive context; the current page is marked.
