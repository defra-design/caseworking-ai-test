Calculations

The page-level pattern a caseworker uses to review the automated rules checks run against a customer's land parcels, find the checks that need attention, and understand what has changed since the last run before approving or rejecting the calculation.

Why we have this page

When an FRPS application is submitted, the service runs a set of automated rules checks against every eligible land-parcel and action combination — for example the required-area check and the moorland-threshold check. A single application can produce hundreds of individual check results across dozens of parcels, and those results can change each time the rules are re-run (because land-cover data is updated, or the customer amends their application).

The caseworker can't act on a flat dump of several hundred results. This page exists to make that volume reviewable: it groups results by parcel and action, collapses each check behind a disclosure so the page stays scannable, surfaces each check's outcome as a tag without the caseworker having to open it, and lets them filter down to just the checks that failed or changed. It is the evidence screen that sits behind the human decision to accept or reject the calculation.

What it's for

  • Reviewing the outcome of the automated rules checks for one application.
  • Focusing quickly on the checks that failed, or that changed since the previous run.
  • Comparing runs — seeing which values moved between two rules-check runs and why a check's result flipped.
  • Providing the auditable detail (the per-check "Check details" tables) that justifies the eventual case decision.

Structure

The page is a three-level hierarchy:

  1. Parcel — a govuk-heading-m, e.g. "Parcel ID: SO3757 3159 checks".
  2. Action — a govuk-heading-s under the parcel, e.g. "Action: CMOR1 checks".
  3. Check — one GOV.UK Details disclosure per check, with a check-result tag in its summary. Expanding it reveals the result, the pass/fail reason and a "Check details" table.

Features

  • Per-check disclosures — each check is a Details component so the page is scannable when collapsed and detailed when opened.
  • Check-result tags — a Passed/Failed (or Checked/Action) tag in every summary row, so the outcome is visible without opening the check.
  • Filter — the calculations filter: check-status views (All checks / Calculations with changes / Only fails) and, on the larger pages, parcel-attribute views (LPs on SSSIs / LPs requiring HEFERs).
  • Run switcher — the filter's Run date dropdown reloads the page at a historical rules-check run, carrying the current filter through.
  • Changed-value highlight — a paler-yellow highlight on the specific value(s) that moved since the previous run.
  • Rerun confirmation — a moj-alert--success banner shown when the calculations have just been re-run.

A single check

The repeating unit of the page. Note the tag sits inside the summary's .govuk-details__summary-text and is aligned to the right automatically by the application-level flex rule (see Details).

Example

Action: CMOR1 checks

Required area check Failed

Result

Failed

Fail reason

The area applied for (4 ha) is less than the total available area (4.53 ha).

Area on moorland check Passed

Result

Passed

Pass reason

Parcel SO3757 3159 is at least 50% on moorland.

Which tag goes where

Two different tag families appear on and around this page — don't mix them up:

  • Inside the page, on each individual check, use the check-result tags — green tick for Passed/Checked, soft red with a cross for Failed, and the pale-blue Action pill where a check still needs follow-up.
  • For the overall case (in the context strip, caselist and review screens), use the status tags — In review, Information requested, Not accepted, and so on.

When to use

  • A caseworker must review a large, repeating set of automated results before making a decision.
  • The results are versioned (re-run over time) and the user needs to see what changed.

When not to use

  • For a short, fixed set of values that doesn't need filtering or disclosures — a plain table or summary list is clearer.
  • To capture the decision itself — that belongs on the review/decision screens.

Accessibility

  • The check outcome is carried by the tag's text (Passed / Failed / Checked / Action), not colour alone.
  • Each check is a native <details>, so it is keyboard operable and its expanded state is announced.
  • Parcel and action headings give the page a real heading outline a screen-reader user can navigate by.
  • The changed-value highlight is deliberately paler than the focus yellow so it isn't mistaken for focus — but don't rely on it alone to signal a change (the "Calculations with changes" filter is the non-visual route to the same information).

Where does this occur

  • Land parcel calculations
  • Real-case calculations (Golden Grange)
  • Large calculations (101 parcels)
  • Medium calculations
