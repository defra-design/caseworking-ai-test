# Structure diff: builder output vs `d2-case-tasklist.js`

Comparing
- **A** — `app/data/builder-tasklist-2026-04-30T13-26-44-217Z.js` (generated by `/builder/new`)
- **B** — `app/data/d2-case-tasklist.js` (hand-authored canonical model)

Both files have the same top-level shape: `{ tasks, stages, decisionOptions }`. The differences are inside each section.

---

## 1. `tasks.<key>`

| Field | A (builder) | B (d2) | Notes |
|---|---|---|---|
| `linkText` | — | ✅ | B drives the visible label on the tasklist; A has no equivalent. |
| `altLinkText` | — | ✅ | B supports per-stage label variants (e.g. `onHold`). Not modelled in A. |
| `pageTitle` | ✅ | — | New in A, captures the H1 of the task page. |
| `preOutcomesHtml` | ✅ | — | New in A, raw HTML body shown above outcomes. |
| `href` | ✅ | ✅ | A auto-generates `<prefix>Task<n>`; B is a free string. |
| `hrefIfFinanceOfficer` | — | ✅ | B supports a conditional href; not modelled in A. |
| `decisionField` | ✅ | ✅ | A: `<prefix>Task<n>Outcome`. B: hand-named (e.g. `decisionTask1`). |
| `acceptedValue` | ✅ | ✅ | A inherits from the stage's `acceptedOutcome`; B set per task. |
| `statusField` | ✅ | ✅ | A auto-generated; B hand-named. |
| `tagField` | ✅ | ✅ | A auto-generated; B hand-named. |

**Key task-key differences**
- A keys are auto-generated (`revTask1`, `agrTask1`).
- B keys are domain-meaningful (`check-customer-details`, `review-land-parcel-calculations`).
- A's task is owned by exactly one stage (the prefix encodes ownership). B's tasks are *reusable* and referenced by multiple stages (`check-customer-details` appears in `start`, `review`, and `on-hold`).

**Key stage modelling difference**
- B separates the **decision options** (in `decisionOptions`) from **what each chosen value does** (the latter lives in `routes.js` lookup tables: `APPROVE_DECISIONS`, `AMEND_OUTCOMES`, etc.).
- A bakes the consequences of each decision into the data itself via `stages.<x>.decisions[]` (`stageChange` + `statusChange`). This means A is more self-contained — the route handler does not need a hand-coded lookup table.

---

## 4. Shape-level summary

| Concern | Builder output (A) | d2 file (B) |
|---|---|---|
| Naming | Mechanical, prefix-based | Domain semantic |
| Task reuse across stages | No — each task belongs to one stage | Yes — tasks referenced from multiple stages |
| Decision consequences (status / stage transitions) | Encoded in data | Encoded in `routes.js` |
| Conditional text variations | Single `conditional` object on stage | Multiple mechanisms: `altLinkText`, `taskLabelVariant`, `showWhen`, separate stages (e.g. `on-hold`) |
| Stage submit / action button | Not modelled | `form` + `action` |
| Per-option visibility rules | Not modelled | `showWhen` |
| Per-task accepted value | Inherited from stage | Set per task |

---

## 5. Gaps to close if A is to drive the same templates as B

To use a builder-generated file in the existing FRPS-D2 templates without losing behaviour:

1. **Task `linkText`** — capture in the task form (currently only page title + body HTML).
2. **`altLinkText` / `taskLabelVariant`** — capture per-stage task label overrides.
3. **`hrefIfFinanceOfficer`** — capture or generalise into a "conditional href" mechanism.
4. **Stage `intro` / `paragraphs` / `links` / `taskListHeading`** — currently collapsed into the single `description` field.
5. **Stage `defaultStatusTag`** — capture default tag class + text per stage.
6. **Stage `form` and `action`** — capture the submit button (text, type, target route).
7. **Decision option `showWhen`** — capture per-option visibility predicates.
8. **Decision `conditionalNoteField` / `conditionalHint`** — currently dropped.
9. **Reusable tasks** — allow a task to belong to more than one stage (today the prefix scheme prevents this).

---

## 6. Where each model is stronger

- **A is stronger** at making decision *consequences* introspectable (no hidden `routes.js` lookup needed) and at enforcing a consistent naming convention.
- **B is stronger** at reusing tasks across stages, at expressing per-option visibility rules, at handling content-only stages (`amend`), and at separating presentation hints (`legendHint`, `conditionalHint`) from option values.
